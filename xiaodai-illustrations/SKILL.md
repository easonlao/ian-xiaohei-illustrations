# 小呆配图 Skill

## 核心定位

当前 skill 的首要职责是：

- 先把小呆角色本体锁稳
- 再输出统一的手绘配图语言
- 在必要时补最小可用的竖版适配

优先级顺序是：

1. 小呆角色稳定
2. 配图结构清楚可用
3. 场景差异只补到够用为止

如果“动作张力”和“角色稳定性”冲突，优先保住角色稳定性。

默认视觉 IP 是“小呆”（小火人形态）：纯手绘亮红色火焰形状实体剪影小人，脸上有一对白色小圆点眼睛，四肢为纤细的黑色火柴人线条。小呆必须参与画面的核心动作，不能只是站在旁边当装饰。

## 先读这些参考

按任务需要读取，不要一次塞满上下文：

- `references/style-dna.md`：统一视觉语言、颜色、留白与非目标风格。
- `references/user-ip.md`：小呆 IP 本体、角色边界与参考图绑定。
- `references/composition-patterns.md`：把抽象内容转成物理动作和低科技隐喻的方法。
- `references/prompt-template.md`：通用提示词模板，等待当前场景传入画幅和场景参数。
- `references/qa-checklist.md`：生成后检查和迭代规则。
- `references/scene-modes.md`：不同场景的最小差异约束。
- `assets/examples/`：只作低频视觉校准，不进入默认生成路径。不要照抄这些案例的构图、物件或标注。

## 工作流

### 1. 先判断内容场景，再判断是否需要竖版适配

先读用户给的正文、链接、Notion 页面、Markdown 文件或截图内容。提炼：

- 核心观点是什么
- 内容主要面向什么消费场景
- 这个场景更偏横图、卡片还是竖屏
- 哪些内容适合用图解释
- 哪些地方只适合文字，不需要图

如果用户没有指定特殊场景，优先采用最稳定、最清楚的正文配图逻辑，不要为了“场景丰富”主动把任务改造成复杂封面策略。

### 2. 先做简单场景判断，再保持清楚的表达骨架

如果用户只给一个主题、一句话观点或一段内容，而没有给完整的 scene spec，不要过度展开成复杂策略。优先做一个简短判断，然后继续沿用最稳定的表达骨架。

优先输出一个很短的 scene decision：

- `scene_mode`
- `format_goal`
- `canvas_ratio`
- `subject_density`
- `structure_type`
- `core_metaphor`

scene decision 的语气要像“这次任务的判断结果”，不是“这个系统永远只能这样”。

例如可以说：

- 这次更适合 `article_inline`
- 推荐 `16:9 horizontal`
- 主体密度偏低，保留大留白
- 结构类型是 `conceptual_metaphor`
- 核心隐喻是“小呆被自己的线团缠住，不是不想动，而是动不了”

字段边界要清楚：

- `scene_mode`：内容要落在哪种消费场景里，例如 `article_inline`、`social_single`、`feed_cover`、`vertical_cover`
- `structure_type`：这张图用什么表达结构，例如 `conceptual_metaphor`、`workflow`、`before_after`、`map_route`
- `format_goal`：这一场景下的阅读目标与构图目标，例如“横向呼吸感段落插图”或“手机信息流里一眼停住的竖屏封面”

`scene_mode` 不要写成 `conceptual_metaphor`、`workflow` 这类结构词。那是 `structure_type`，不是场景。

优先从下面这些 `scene_mode` 里选：

- `article_inline`：长文正文中的段落配图
- `social_single`：单张观点配图，可用于文章内嵌或社交传播
- `feed_cover`：信息流封面，强调一眼停住
- `vertical_cover`：竖屏封面，强调手机首屏张力

如果确实不在这 4 类里，也要先回答“这张图被放在哪里被看见”，再决定 scene_mode，而不是退回结构词。

如果同一个主题要跨两个场景输出，先固定核心隐喻，再分别做最小改动：

- 视角远近
- 留白比例
- 信息密度
- 标注数量
- 画面张力

如果用户只是说“怎么配图 / 这个 IP 怎么用于某种内容 / 先出策略”，先输出：

- 推荐场景
- 推荐画幅
- 推荐主体占比
- 推荐文字密度
- 小呆在图里承担什么动作
- 建议使用什么低科技隐喻

如果任务明确是长文配图，再继续输出 shot list。每张图写清楚：

- 图的主题
- 核心意思
- 结构类型
- 小呆在图里做什么
- 建议元素
- 建议中文标注词

一旦 `scene_mode` 已经确定，继续读取 `references/scene-modes.md`。默认原则仍然是：优先保住稳定清楚的配图行为，只在必要处补差异。

不要只给比例，还要带入：

- 推荐构图重心
- 推荐留白位置
- 推荐主体高度或占比
- 推荐标注数量
- 推荐张力目标
- 该场景下最容易失败的点

### 3. 生成时先保住角色和清楚表达，再谈场景差异

如果用户明确要求"生成 / 输出 / 做图 / 帮我生成"，不要停下来等确认；用内置 `image_gen` 每张单独生成。不要把多张图拼在一张里。

每张图只讲一个核心结构。生图时优先使用 `references/user-ip.md` 中的标准参考图，确保 IP 特征一致。

角色一致性优先级高于构图创意、动作复杂度和场景张力。只要出现“更有张力但更不像小呆”和“更像参考图但略保守”之间的冲突，永远选后者。

如果用户没有给完整 scene spec，生成前先用 3-5 行文字显式输出你本次采用的：

- `scene_mode`
- `format_goal`
- `canvas_ratio`
- `subject_density`
- `structure_type`
- `core_metaphor`

然后再写最终 prompt。不要直接从一句主题跳到一大段 prompt，中间没有任何场景判断。

提示词必须包含：

- 用户指定或当前判断出的场景说明
- 用户指定或当前判断出的画幅说明
- 用户指定或当前判断出的主体密度
- 对应 scene mode 的构图约束
- 参考图优先于创意变化的硬约束
- 纯白背景
- 黑色手绘线稿
- 少量红色/橙色/蓝色中文手写批注
- 大量留白
- 小呆作为核心动作主体
- 禁止 PPT、商业插画、幼稚可爱、复杂架构、左上角类型标题

不要机械复刻过往案例，但也不要为了“全新”而牺牲解释力。优先保持图能一眼讲清结构，再把小呆放进核心动作里。

特别注意：

- `16:9` 可以是这一次长文场景的正确结论，但不能写成系统默认真理。
- `3:4` 可以是封面场景的正确结论，但不能反过来写成所有任务都必须竖屏。
- 这套 skill 不是要把每个 scene mode 都发展成独立系统。
- 如果小呆开始长出独立躯干、可爱表情或 mascot 感，不要继续堆更多解释；先缩减动作复杂度，回到更安全的姿态。

### 4. 检查与迭代

生成后检查 `references/qa-checklist.md`。如果出现以下问题，优先重生成或局部编辑：

- 小呆只是装饰
- 画面太满
- 太像流程图/PPT
- 中文太多或错字严重
- 左上角出现"常见坑/流程图/系统架构图"等标题
- 画风太可爱、幼稚、死板
- 背景不是干净白底

### 5. 保存交付

如果用户在 workspace 内工作，把最终图复制到：

```text
outputs/<article-slug>-illustrations/
```

按顺序命名：

```text
01-topic-name.png
02-topic-name.png
```

保留原始生成文件，不要覆盖已有输出，除非用户明确要求替换。

## 输出口径

生成前的策略输出要短而准。生成后的交付要包含：

- 使用了什么场景判断
- 使用了什么画幅判断
- 生成了几张
- 每张图的用途
- 保存路径
- 哪些图最稳，哪些图是可选

不要长篇解释风格理论；让图自己说话。
